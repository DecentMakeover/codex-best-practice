Skills guide

Package a workflow, not a vibe.

If you keep pasting the same long prompt or correcting the same behavior, the workflow probably wants to become a skill.

Use a skill when

  • The task repeats often enough to deserve structure
  • The inputs and outputs are easy to describe
  • You want Codex to follow the same steps every time
  • Extra references or scripts improve reliability

Keep the description sharp

Codex uses the skill description to decide when the workflow applies. Write it like a boundary, not a slogan. 

---
name: release-notes
description: Draft release notes from merged pull requests when the user asks
for a changelog, weekly summary, or release draft. Do not use for general docs.
---

1. Gather the merged PR context.
2. Group changes by user-facing impact.
3. Call out risky or breaking changes.
4. Return a draft with clear section headings.

When to add more than instructions

Start instruction-only. Add scripts only when deterministic behavior or external tooling clearly improves the result. Add MCP dependencies when the workflow needs live context.
