config.toml guide

Choose defaults that reduce negotiation.

Good Codex config is mostly about consistency: the same model behavior, the same safety posture, and the same tooling available every time someone opens the repo.

Start with these defaults

  • A default model and reasoning effort
  • An approval policy the team is comfortable with
  • A sandbox mode that matches the trust boundary
  • Only the MCP servers and apps that unlock real work

Typical repo split

Keep personal defaults in ~/.codex/config.toml. Keep repo-specific behavior in .codex/config.toml. Use command-line flags for one-off exceptions, not policy. 

model = "gpt-5-codex"
approval_policy = "on-request"
sandbox_mode = "workspace-write"
web_search = "cached"

[features]
multi_agent = true

[agents]
max_threads = 6

[mcp_servers.openaiDeveloperDocs]
url = "https://developers.openai.com/mcp"

Document the tradeoffs

If you choose broader permissions or enable more tools, explain why. The goal is not “maximum autonomy,” it is a repeatable level of autonomy your team understands.
